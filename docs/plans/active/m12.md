## Entry Animation Expansion (scoping, pre-implementation)

The site currently reads as flat because entry animation is effectively limited to two hand-built cases (`home-hero-carousel.tsx` slide transitions and the pet detail image gallery's fade-in), while hover motion is comparatively well-covered. This section scopes closing that gap with subtle fade/slide-up-on-entry motion, site-wide. Not yet implemented — outline only.

Relevant precedent from `docs/dependency-decision-log.md:38-39`: Tailwind/CSS-only animation is the default, and Framer Motion is the "preferred escalation path... if scroll-triggered patterns become common," with an explicit warning to **not build a broad custom Intersection Observer animation framework before considering Framer Motion instead**. Since this expansion is exactly a site-wide scroll-triggered pattern, implementation should treat the Framer Motion decision as live to revisit rather than defaulting to a hand-rolled observer hook.

Candidate checklist for implementation (not started):

- [ ] Decide the animation approach: adopt Framer Motion for scroll-triggered entry animation (per the decision log's own escalation criteria), or extend the existing CSS-keyframe + `motion-safe:` pattern (`app/globals.css:119-156`, e.g. new `*-enter` keyframes) with a small shared Intersection Observer hook. Record the decision in `docs/dependency-decision-log.md` either way.
- [ ] If a new dependency is added, update `ATTRIBUTIONS.md` and confirm license terms per the decision log's existing placeholder note ("verify current package and license before install").
- [ ] Add entry animation to page-builder section renderers in `components/features/sections/page-sections.tsx` (hero, hero slide, content section, callout, stat block, feature list, accordion, pricing sections, warranty sections, process path, CTA group) — fade/slide-up as each section enters the viewport.
- [ ] Add entry animation to the pet marketplace grid (`app/(site)/pets/page.tsx` grid wrapper and/or `pet-card.tsx`) so cards animate in on initial load and on filter-change, instead of snapping in.
- [ ] Confirm all new entry animation respects the existing global `prefers-reduced-motion` block (`app/globals.css:147-156`) or uses `motion-safe:` variants consistently with the rest of the codebase.
- [ ] Keep motion subtle (fade + small slide-up, short duration) consistent with the existing `pet-hero-slide-enter`/`pet-gallery-image-enter` keyframe timings (~120-260ms) rather than introducing a different motion language.
- [ ] Re-run the Milestone 12 responsive/overflow pass after adding animation, since transform/opacity transitions on scroll can interact with layout in ways a static read can't catch.
