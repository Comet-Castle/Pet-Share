## Entry Animation Expansion (scoping, pre-implementation)

The site currently reads as flat because entry animation is effectively limited to two hand-built cases (`home-hero-carousel.tsx` slide transitions and the pet detail image gallery's fade-in), while hover motion is comparatively well-covered. This section scopes closing that gap with subtle fade/slide-up-on-entry motion, site-wide. Not yet implemented — outline only.

Relevant precedent from `docs/dependency-decision-log.md:38-39`: Tailwind/CSS-only animation is the default, and Framer Motion is the "preferred escalation path... if scroll-triggered patterns become common," with an explicit warning to **not build a broad custom Intersection Observer animation framework before considering Framer Motion instead**. Since this expansion is exactly a site-wide scroll-triggered pattern, implementation should treat the Framer Motion decision as live to revisit rather than defaulting to a hand-rolled observer hook.

Candidate checklist for implementation:

- [x] Decided the animation approach: extended the existing CSS-keyframe + `motion-safe:` pattern with a small shared Intersection Observer hook (`lib/hooks/use-in-view.ts`) plus a `Reveal` wrapper (`components/ui/reveal.tsx`). No Framer Motion added. Decision recorded in `docs/dependency-decision-log.md` (Framer Motion row).
- [x] No new dependency added, so `ATTRIBUTIONS.md` needs no change for this work.
- [x] Added entry animation to page-builder section renderers via `PageSections` (`components/features/sections/page-sections.tsx`) — every section past the first (above-the-fold) fades/slides up as it scrolls into view.
- [x] Added entry animation to the pet marketplace grid (`app/(site)/pets/page.tsx`) and the homepage featured-pet grid + major homepage sections (`app/(site)/page.tsx`). Cards use a per-row stagger (`index % 3`) and re-reveal on filter/pagination navigation instead of snapping in. Verified in-browser: below-fold cards start `opacity:0` and reach `opacity:1` after scroll.
- [x] Confirmed reduced-motion handling: `Reveal` gates all transform/opacity behind `motion-safe:`, so `prefers-reduced-motion` users see content immediately (verified in-browser: reduced-motion card is `opacity:1` with no scroll).
- [x] Motion kept subtle (fade + `translate-y-4` slide-up, 500ms `ease-out`, capped stagger) consistent with the existing motion language.
- [x] Re-ran a responsive/overflow spot check after adding animation (desktop pets/home screenshots): no overflow, clipping, or overlap introduced by the transitions.
