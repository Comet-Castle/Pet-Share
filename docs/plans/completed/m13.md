## Framing

The seed tooling (`scripts/seed-wizard.mjs`, `scripts/seed-sanity.mjs`,
`scripts/seed-media.mjs`, `scripts/seed-env.mjs`) is a single-operator, internal-only
tool тАФ it is not shared infrastructure and nothing outside this repo depends on it.
This milestone is a **light usability pass**, not a redesign: better visibility into
what the tooling is doing while it runs, and the ability to seed specific quantities
of pets, owners, and testimonials without touching code. It replaces the earlier
"three explicit phases" framing from the original M13 draft, which added more
ceremony (phase menus, guard prompts, dangling-ref warnings) than this tool's actual
stakes justify.

Two things this milestone explicitly does **not** do:

- It does not restructure `seed-wizard.mjs`'s quick-replace / step-wizard modes into
  separate phases. The existing modes stay as they are.
- It does not add progress output to synchronous, in-memory steps (e.g. page-builder
  section construction in `buildPages`/`transformPage`/`transformSection`). Those
  steps run in milliseconds regardless of content volume тАФ there is no real wait to
  visualize, so progress output there would be noise, not signal.

## Configurable quantities

- [x] **Add an `--owner-count` flag** (and matching wizard prompt) alongside the existing `--pet-count`. `ownerSeeds` is a fixed 12-entry list (`scripts/seed-sanity.mjs`) with no overflow-generation pattern today, unlike pets. **Decision: no synthetic owner generation in this milestone.** `--owner-count` selects a prefix of the fixed list (`ownerSeeds.slice(0, count)`), clamped to a max of 12. Requesting more than 12 prints a warning and clamps rather than silently doing nothing or inventing owners. Generating additional synthetic owners (bios, taglines, locations) is real content-generation work, not a quantities/progress change тАФ note it as explicitly deferred, not built here.
- [x] **Add a `--testimonial-count` flag** (and matching wizard prompt), scoped only to the **generated** (pet-derived) testimonial slice in `buildTestimonials` (`scripts/seed-sanity.mjs` ~line 1169), not the authored set. `buildTestimonials` always includes every item from `sanity/seed/data/testimonials.json` in full тАФ those are curated content, not volume filler, and stay untouched. **Decision:** `--testimonial-count` caps the generated slice at `min(count, pets.length, 18)` (18 is today's hardcoded cap via `pets.slice(0, 18)`). Document in code and README that the flag affects generated testimonials only; total testimonial count in output = authored count + generated count.
- [x] **Confirm `homePage.testimonialSeedIds`/`testimonialSeedKeys` stay valid when the generated testimonial slice shrinks.** `transformPage`'s homePage branch slices `testimonials.filter((item) => item.featured).slice(0, 6)` тАФ verify this still resolves correctly (no dangling seed keys) when `--testimonial-count` reduces the generated pool below what featured-selection expects.
- [x] **Keep pages, forms, and site settings fixed-size.** These are not repeated per-item content (one homepage, one set of system pages, a fixed handful of marketing pages and forms) тАФ no count flag needed for them.
- [x] **Update preview summary output** (`writePreviewFiles` in `seed-sanity.mjs`) to report the actual owner count and the authored/generated testimonial split, the same way it already reports `actualPetCount`.
- [x] **Update the wizard's `askPetCount`-equivalent prompts** in `scripts/seed-wizard.mjs` to also ask for owner and testimonial counts where relevant, without restructuring the wizard's existing modes.

## Batching

- [x] **Parallelize `uploadKnownAssets` using a hand-rolled chunked-parallel loop тАФ no new dependency.** It currently uploads discovered media files strictly one at a time via a `for...of` loop. Reuse the existing `chunkArray` helper already used for document mutations: split the file list into small chunks (e.g. 5тАУ10 files), `Promise.all` the uploads within each chunk, and report progress as each upload in the chunk settles (not all at once at the end). No concurrency-limiting package exists in `package.json` today; per `CLAUDE.md`'s dependency guidance, a small hand-rolled chunk loop is preferable to adding one (e.g. `p-limit`) for this.
- [x] **Confirm existing document mutation batching stays effective.** No change made тАФ default pet/owner/testimonial counts are unchanged, so chunk counts (25/50) stay the same as before. `upsertBatch`/`upsertSingletons`/`purgeSeedDocuments` already chunk into groups of 25/50 via `chunkArray` тАФ no change needed unless raising pet/owner/testimonial counts significantly increases chunk counts in a way worth tuning. Note if chunk sizes should change as a result of the new count flags.
- [x] **Do not attempt to batch Gemini media generation calls.** `generateWithGemini` in `seed-media.mjs` is necessarily sequential (one request per image, human-confirmed, quota-sensitive) тАФ leave this loop as-is. Batching here is out of scope and would risk unwanted provider load.
