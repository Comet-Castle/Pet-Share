## SEO, metadata, and Open Graph (implementation)

Much of the SEO system already exists and should be reused, not rebuilt: the
tested `lib/content/metadata.ts` `metadataFromSeo()` helper (page → site-default →
static `public/og-default.png` OG fallback, PNG normalization at 1200×630,
canonical/`og:url`, Twitter card, `noIndex` → robots) is wired into every route's
`generateMetadata`; the Sanity `seo` object and `siteSettings.defaultSeo` exist;
and the seed script generates per-document `seo` for pets, owners, the pet index,
and marketing pages. This section closes the remaining gaps so M16 stays pure
deployment.

- [x] **Wire `siteDefaultSeo` into every route.** `metadataFromSeo()` accepts and is unit-tested for `siteDefaultSeo`, but no route currently passes it — so the CMS-authored `siteSettings.defaultSeo` OG image is never used and pages without their own OG fall straight to the static PNG. Load `siteSettings.defaultSeo` (already fetched in `sanity/queries/settings.ts`) and pass it through in home, `/pets`, pet detail, owner detail, and marketing routes.
- [x] **Add `app/sitemap.ts`.** Generate a sitemap from the live dataset: static routes plus all pet slugs, owner slugs, and marketing-page slugs. Respect `noIndex` and exclude Studio/API/draft routes. Use `NEXT_PUBLIC_SITE_URL` for absolute URLs.
- [x] **Add `app/robots.ts`.** Allow public routes, disallow `/studio`, `/api`, and draft-mode routes, and reference the sitemap URL.
- [x] **Build composed dynamic OG images for pets and owners.** Decision (2026-07-08): don't just reuse the raw photo — generate a designed share card. Add `opengraph-image.tsx` (Next.js `ImageResponse`, 1200×630) for both `app/(site)/pets/[slug]/` and `app/(site)/owners/[slug]/`. Layout: the subject's **hero shot on the left**, and on the **right** the Pet Share logo plus key details — name, a couple of defining attributes (e.g. pet: breed/type; owner: role/location), and a brief one-line description. Keep the two cards visually consistent (shared template/helper). Fetch only the fields the card needs from Sanity (hero image URL, name, summary) and handle the missing-image case gracefully.
- [x] **Reconcile the dynamic OG image with `metadataFromSeo`.** Decision (2026-07-08): pet and owner detail pages use their composed route-level card as the social image so shared listings/profiles have consistent branded previews. CMS-authored OG images and `siteSettings.defaultSeo` remain the fallback chain for home, `/pets`, and marketing pages; `public/og-default.png` stays the last-resort fallback.
- [x] **Per-route metadata resolves end-to-end** against the final dataset: every route produces a sensible title/description, canonical URL, and OG/Twitter image from Sanity with correct fallback order, and no OG image is a broken reference.
- [x] **Create the favicon and app icons.** None exist today — no `app/icon.*`, `app/favicon.ico`, `app/apple-icon.*`, or manifest — so the site serves the Next.js default. Add a branded favicon (prefer `app/icon.svg` for the App Router auto-`<link>` convention, plus `app/apple-icon.png` for iOS and an `.ico` fallback for legacy browsers). Optionally add `app/manifest.ts` (name, theme color, icons) for PWA/install metadata. If the icon is produced via a metered AI provider, generation is human-run per the AI Media Generation rules; a simple hand-authored brand SVG is preferable here anyway.
- [x] **Sitewide metadata** in `app/layout.tsx` (site name, title template, `metadataBase`, default social image) is correct for launch, and the new icons resolve on every route.
- [x] **Validate social cards** with a scraper/validator (or at minimum confirm the resolved `<meta>` tags and that image URLs return 200 PNGs at 1200×630) for a representative page of each type.
- [x] **Add tests for new SEO logic** where feasible, matching the existing coverage on `lib/content/metadata.test.ts`: sitemap URL/entry generation, and any pure data-assembly helper backing the composed OG card (e.g. the function that picks title/attributes/description for the card, separate from the `ImageResponse` rendering itself).

## Specific issues found during pre-M15 review

Concrete items already diagnosed (do not implement until M15 is active). Each notes
the cause and intended fix so the work is unambiguous when the milestone starts.

- [x] **Pricing value section collapses to a vertical list too late.** The "Owners list / Hosts earn / Everyone agrees" row (`PricingValueSection` in `components/features/sections/page-sections.tsx`) uses `sm:grid-cols-3`, so it goes 3-across at ~640px and is cramped on tablet. Change it to stay a single vertical column through tablet and only go 3-across at `lg` (~1024px).
- [x] **Process-path header icons crowd small screens.** On the how-it-works page, the section-level icons (house for "For owners sharing their pet", magnifying glass for "For temporary hosts") in `ProcessPathSection` (`page-sections.tsx`) should be hidden on smaller screens — they're decorative (the per-step icons are separate), so gate the icon `<span>` to `hidden … sm:flex`. Verify during the marketing-page review at mobile/tablet/desktop.
- [x] **Navigating between pages lands scrolled past the header.** Root cause: `html { scroll-behavior: smooth }` in `app/globals.css` interferes with the App Router's scroll-to-top on navigation, leaving the new page partway down with the header off-screen. Fix by removing the global smooth rule (and its now-redundant reduced-motion override). Trade-off: the pet-detail anchor sub-nav (`#facts`, `#about`, …) then jumps instantly; restoring smoothness without reintroducing the bug needs a small JS handler scoped to those anchor clicks.
- [x] **Sanity Studio shows "Invalid property value" on many docs.** Code fix landed and production was purge re-seeded with the null-stripping fix. Typed object fields (e.g. `cta`, `ctaGroup`, `primaryCta`) are stored as explicit `null`, which Sanity rejects (absent = key omitted, never `null`). Source: literal `"cta": null` entries in `sanity/seed/data/pages.json` plus seed transform helpers that `return null` for absent objects. Fix: strip `null`/`undefined` keys in `resolveReferences` (`scripts/seed-sanity.mjs`) — the single recursive chokepoint every doc passes through before writing. Note: the normal write uses `patch.set`, which won't unset already-stored nulls, so cleaning the live dataset requires a `pnpm seed:sanity -- --confirm --purge` re-seed.
- [x] **Pet index sort options.** Reduce the marketplace sort options to **Featured · Newest · Distance · Alphabetical**, removing "Chaos: low to high", "Chaos: high to low", and "Soonest pickup". Touches `petIndexSortOrders` in `sanity/queries/pets.ts` (add `distance` via `defined(distanceKilometers) desc, distanceKilometers asc, …` so pets with no distance sort last, and `alphabetical` via `name asc`) and `petIndexSortOptions` in `components/features/pets/pet-index-state.ts`. `featured` stays the default.
